1 Listed herein are language incompatibilites between 3.3 and 3.5,
2 that you should pay attention to when updating.
3 
4 - LPC features
5  + structs are always supported.
6  __LPC_STRUCTS__ is always defined.
7 
8  - inline closures are always supported.
9  __LPC_INLINE_CLOSURES__ is always defined.
10  - call-other on arrays is always supported.
11  __LPC_ARRAY_CALLS__ is always defined.
12  - alists are no longer supported.
13  __ALISTS__ was removed.
14 - sefun prefix:
15  The 3.5 driver allows to explicitly call simul-efuns using a 'sefun::'prefix, e.g. sefun::fun(). If a program has inherited anotherprogram with the name "sefun" and explicitly called inheritedlfuns with that syntax, this won't work anymore, because thedriver now tries to find a simul-efun with that name.This also applies to symbol_function and closure literals (#'sefun::fun).
16 - ptmalloc keywords for debug_info():
17  DID_MEM_MMAP, DID_MEM_MMAP_SIZE, DID_MEM_FREE_CHUNKS, DID_MEM_FFREE,DID_MEM_FFREE_SIZE, DID_MEM_MAX_ALLOCATED and DID_MEM_KEEP_COST wereremoved.
18 - Removed efuns:
19  + The deprecated efuns cat() and tail() were removed. They can be
20  easily and more flexibly replaced by sefuns, which are supplied with
21  the driver in /mudlib/deprecated/. (#637, #228)
22 
23  - The efun set_light() was removed. This includes the internal light
24  state managment in the driver. The mudlib does a better and moreflexible job at managing light in/from objects. (#775)
25  - The supporting efuns for alists were removed: insert_alist, order_alist,
26  intersect_alist, assoc.
27 - New efuns:
28  + json_parse() parses strings encoding a JSON object into an
29  appropriate LPC value. (optional efun)
30 
31  - json_serialize() converts a LPC value into an JSON object and
32  serializes it as LPC string. (optional efun)
33 - Changed efuns:
34  + call_other() and call_direct() now always accept an array of objects or
35  strings (object names) as argument and call the function in all of these
36  objects.
37 
38  - object_info() does not report OIB_TOTAL_LIGHT any more, therefore the
39  array OINFO_BASIC is one element shorter.
40 - Pragmas
41  + The behaviour of the pragma combine_strings is mandatory, that means
42  string literals concatenated by '+' will always be combined at
43  compile-time. This pragmas and no_combine_strings are ignored. (#702)
44 
45  - The behaviour of the pragma local_scopes is mandatory, that means
46  that variables defined in a block always cease to exist after the end ofthe block. This pragma and no_local_scopes are ignored (#702)
47  - The behaviour of the pragma verbose_errors is mandatory, the compiler will
48  always give information about the context of an error.This pragma is ignored (#702)
49  - The pragma warn_deprecated is now enabled by default. (#702)
50 - 'deprecated' modifier
51  The deprecated modifier can be used for lfuns (and sefuns) and globalvariables. Upon creating and closure to them or using/calling them thecompiler will issue a warning. The same occurs in symbol_function(),symbol_variable() and call_other/call_direct/call_resolved().
52 - LPC types
53  + the 'float' type on 64 bit platforms (specifically LP64) has a higher
54  precision and larger value ranges, because the driver uses the native
55  'double' types of the host system to store them. On most host systems they
56  conform to the IEEE 754 double precision floating point format.
57  WARNING: If you want to save them losslessly with save_object/save_value(),
58 
59  you MUST use savefile format version 2 or higher!
