1 Intermud communication {#driver_topics_intermud}
2 ================================================
3 There are several intermud protocols which define how (players on) different muds can communicate with each other. The protocols are in general not muddriver or mudlib dependant, though the number of implementations is limited.
4 
5 This text is about the rather old widely spread 'Zebedee Intermud', which is also called 'Intermud 2' although it differs quite a lot from the real Intermud 2 protocol.
6 
7 Full information on the newer Intermud 3 could be found on the web at @subpage driver_http:__www.imaginary.com_protocols_intermud3.html "http://www.imaginary.com/protocols/intermud3.html" so there is no discussion here - the following is just about Zebedee Intermud (aka Intermud 2).
8 
9 Zebedee Intermud communication is handled by the /secure/inetd object, originally written by Nostradamus for Zebedee with some extensions that are discussed in >>:concept:`inetd`<<. How the data is actually sent across the network is described in intermud.basic(C).
10 
11 ## Services #
12 
13 Note that the fields "NAME" and "UDP_PORT" should be present in every message. Very common are the fields "ID" (used whenever an reply is expected) and "SND" (the sender: he should receive the reply). These fields will not be mentioned in the list below.
14 
15 Request types are listed on the leftmost row (e.g. "REQ=channel"), associated header are listed indented.
16 
17 "channel"
18 The channel-request is used for sending a message on any channel. The "CMD" field is optional and may be omitted for normal messages. Note that you should not send an history or list request to _all_ known muds!
19 
20 "CHANNEL"
21  The channel on which a message is send (the standard channels are "intermud", "intercode", "interadm", "d-chat", "d-code" and "d-adm"; on the d-channels German is spoken)
22 
23 "DATA"
24  The message to be send (not used with history/list request)
25 
26 "CMD"
27  The body of this header may be:
28  "" for normal intermud messages,
29  "emote" if the message is an emote/gemote,
30  "history" for an history request: the last 20 lines of this channel will be shown.
31  "list" to list all remote users listening to this channel
32 
33 "EMOTE" (optional)
34  The body is 1 if the message is an emote.
35  The body is 2 if the message is a gemote.
36 
37 "finger"
38 Retreive information about a player or creator on a remote mud.
39 
40 "DATA"
41  The player of whom information is requested
42 
43 "locate"
44 Check whether a certain player is logged on at a remote mud. This request is usually send to all known muds at the same time.
45 
46 "user"
47  Name of the person who requests the information. This is used by the sending mud only and has to be included in the reply.
48 
49 "vbs"
50  The verbose option has only two pre-defined values:
51 
52  1:
53  Even report when the result was negative
54 
55  2:
56  Don't do timeouts, but keep waiting
57 
58  This is used by the sending mud only and has to be included in the reply.
59 
60  "fnd"
61  The found option is only used in the reply and it's value is either 1 (success) or 0 (failure). The absence of a found parameter indicates failure as well.
62 
63  "DATA"
64  The player to find.
65 
66 "man"
67 Retreive a manual page from a remote mud. Many muds don't support this feature...
68 
69 "DATA"
70  The name of the requested manual page
71 
72 "mail"
73 An extension to the standard protocol, by @subpage driver_mailto:Alvin@Sushi "Alvin@Sushi". This is used to send mails from one mud to another.
74 
75 "udpm_status"
76  This field should only be used in the reply and indicates how mail is handled. Currently there are four pre-defined values for the status field:
77  0 time out
78  1 delivered ok
79  2 unknown player
80  3 in spool (will be delivered later)
81 
82 "udpm_writer"
83  Name of the person who wrote this mail
84 
85 "udpm_spool_name"
86  Should be returned as sent, this value is used to remove the mail from the spool directory after it has been delivered (or refused)
87 
88 "udpm_subject"
89  Subject of the mail message
90 
91 "DATA"
92  The body of the mail (the actual message)
93 
94 "ping"
95 A ping request has only the standard fields, the reply is usually a short string like " is alive."
96 
97 "query"
98 Get standard information about another mud. This is the only command of which the reply may not include a load of rubbish, but should only hold the requested information, so that it can be parsed by the server.
99 
100 "DATA"
101  The following queries are pretty much standard:
102  "commands" List all commands that are supported by the inetd
103  "email" The email-address of the mud administrator(s)
104  "hosts" A listing of all hosts in a special format [t.b.d.]
105  "inetd" The version number of the inetd used
106  "list" The list of all items which can be queried
107  "info" A short human-readable string with practically
108 
109  "query" information
110 
111  "mud_port" The portnumber that players connect to on login
112  "time" The local time for this mud
113  "users" A list of the people that are active in this mud
114  "version" The version of the mud-driver (and library)
115  "www" The URL of the mud's web page (e.g.
116 
117  @subpage driver_http:__mud.stack.nl_ "http://mud.stack.nl/")
118 
119 "reply"
120 This request method is used for _all_ replies.
121 
122 "DATA"
123  A human-readable string, containing the reply to a given query
124 
125 "RCPNT"
126  The same name as in the "SND" field or the query; Usually this is the name of the player who initiated the query
127 
128 "QUERY"
129  This field is only used in a response to a "query" request and should be equal to the "DATA" field of that request
130 
131 "vbs"
132  This field is only used in a response to a "locate" request and should be equal to the "vbs" field of that request
133 
134 "user"
135  This field is only used in a response to a "locate" request and should be equal to the "user" field of that request
136 
137 "fnd"
138  This field is only used in a response to a "locate" request and should be 1 if the player was located and 0 otherwise
139 
140 "tell"
141 Say something to a player on another mud.
142 
143 "RCPNT"
144  Name of the player to whom you are talking
145 
146 "DATA"
147  Whatever you wish to say to this person
148 
149 Optional emote-tos are handles are also handled as tells, so muds without emote-to support display them as reasonable readable tell message.
150 
151 "RCPNT"
152  Name of the player to whom you are talking
153 
154 "METHOD"
155  The body of this header may be:
156  "emote" if the message is an emote
157  "gemote" if the message is a genitiv emote
158 
159 "DATA"
160  The text to be emoted prepended with "*" and appended with "* ". If you display the emote you have to cut the stars off. Muds that do not process emote-tos display the emote as tell message with the stars as indication of the message's emote meaning.
161 
162 "who"
163 List the people that are active on a remote mud. The anwer usually contains some active information about the players, like titles, levels or age.
164 
165 "DATA"
166  Not supported by many muds. Introduced August 1997. Additional switch(es) (blanc separated) that change the appearence of the resulting list. The switches normally resemble the switches used inside of that mud for the 'who' command. Typical values include:
167  "short" "s" "-short" "-s" "kurz":
168 
169  Return a concise listing.
170 
171  "alpha" "a" "alphabetisch" "-alpha" "-a"
172  Sort the players alphabetically.
