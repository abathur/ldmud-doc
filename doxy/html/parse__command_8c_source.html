2  /**
3  * parse_command() is basically a spiffed up sscanf operating
4  * on word basis and targeted at recognizing object descriptions from
5  * command strings.
6  *
7  * The efun takes the command string <cmd> and the object(s) <env>/<arr>
8  * and tries to match it against the format string <fmt>. Successfully
9  * matched elements are assigned to the variables <var>.... The result
10  * from the efun is 1 if the command could be fully matched, and 0
11  * otherwise.
12  *
13  * If the objects are given as a single object <env>, the efun matches
14  * against the given object and all objects contained therein. Otherwise,
15  * if the objects are given as an array <arr> of objects, the efun
16  * matches only against the given objects.
17  *
18  * The format string <fmt> consists of words, syntactic markers, and
19  * %-directives for the values to parse and return in the variables.
20  * A typical example is " 'get' / 'take' %i " or
21  * " 'spray' / 'paint' [paint] %i ". The elements in detail are:
22  *
23  * 'word': obligatory text
24  * [word]: optional text
25  * / : Alternative marker
26  * %o : Single item, object
27  * %s : Any text
28  * %w : Any word
29  * %p : One of a list of prepositions.
30  *
31  * If the variable associated with %p is used to pass
32  * a list of words to the efun, the matching will take
33  * only against this list.
34  *
35  * %l : non-compat: Living objects
36  * compat: a single living object
37  *
38  * %i : Any objects
39  * %d : Number >= 0, or when given textual: 0-99.
40  *
41  * A <word> in this context is any sequence of characters not containing
42  * a space. 'living objects' are searched by calls to the (simul)efuns
43  * find_player() and find_living(): both functions have to accept a name
44  * as argument and return the object for this name, or 0 if there
45  * is none.
46  *
47  * The results assigned to the variables by the %-directives are:
48  *
49  * %o : returns an object
50  * %s : returns a string of words
51  * %w : returns a string of one word
52  * %p : if passed empty: a string
53  *
54  * if passed as array of words: var[0] is the matched word
55  *
56  * %i : returns an array with the following content:
57  * [0]: int: the count/number recognized in the object spec
58  * > 0: a count (e.g. 'three', '4')
59  * < 0: an ordinal (e.g. 'second', 'third')
60  * = 0: 'all' or a generic plural such as 'apples'
61  *
62  * [1..]: object: all(!) objects matching the item description.
63  * In the <env> form this may be the whole
64  * recursive inventory of the <env> object.
65  *
66  * It is up to the caller to interpret the recognized numeral
67  * and to apply it on the list of matched objects.
68  *
69  * %l : non-compat: as %i, except that only living objects are
70  * returned.
71  *
72  * compat: as %o, except that only a living object is returned.
73  *
74  * %i (and non-compat-%l) match descriptions like 'three red roses',
75  * 'all nasty bugs' or 'second blue sword'.
76  *
77  * Note: Patterns of type: "%s %w %i" might not work as one would expect.
78  * %w will always succeed so the arg corresponding to %s will always be
79  * empty.
80  *
81  * To make the efun useful it must have a certain support from the
82  * mudlib: it calls a set of functions in objects to get the
83  * information it needs to parse a string.
84  *
85  * 1. string >>*<<parse_command_id_list()
86  * Normal singular names of the object.
87  *
88  * 2. string >>*<<parse_command_plural_id_list() - optional
89  * Plural forms of the names returned by 1.
90  * If this function doesn't exist, the parser tries to pluralize
91  * the names returned by 1.
92  *
93  * 3. string >>*<<parse_command_adjectiv_id_list() - optional
94  * All adjectives associated with this object.
95  *
96  * All names and adjectives may consist of several words separated
97  * by spaces.
98  *
99  * These functions should exist in all objects and are therefore best
100  * put into a mandatory inherit file (e.g. /std/object.c).
101  *
102  * In addition the master object may offer the same functions to provide
103  * reasonable defaults (like 'thing' as generic singular name):
104  *
105  * string >>*<<parse_command_id_list()
106  * - Would normally return: ({ "one", "thing" })
107  *
108  * string >>*<<parse_command_plural_id_list()
109  * - Would normally return: ({ "ones", "things", "them" })
110  *
111  * string >>*<<parse_command_adjectiv_id_list()
112  * - Would normally return ({ "iffish" })
113  *
114  * Two additional functions in the master object provide the default
115  * list of prepositions (needed for %p) and the single 'all' word:
116  *
117  * string >>*<<parse_command_prepos_list()
118  * - Would normally return: ({ "in", "on", "under", "behind",
119  * "beside" })
120  *
121  * string parse_command_all_word()
122  * - Would normally return: "all"
123  *
124  * int parse_command(string, object|object*, string, destargs...)
125  *
126  * @synopsis{
127  * int parse_command(string cmd, object  env, string fmt, mixed &var, ...)
128  * int parse_command(string cmd, object *arr, string fmt, mixed &var, ...)
129  * }
130  *
131  * @usage{
132  * object *items;
133  * parse_command("take apple",environment(this_player()),
134  * " 'get' / 'take' %i ", items);
135  * }
136  *
137  * @history{
138  * changed (3.3.258) -- removed the compat-mode parse_command().
139  * }
140  *
141  * @see efun::sscanf()
142  **/
143  int parse_command(string cmd, object *arr, string fmt, mixed &var, ...);
144 }
