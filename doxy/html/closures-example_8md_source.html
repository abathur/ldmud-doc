1 closures example {#driver_syntax_closures-example}
2 ==================================================
3 This document contains small examples of the usage of (lambda-)closures. For technical details see the @ref driver_syntax_closures "closure" doc. For hints when to use which type of closure, see the end of this doc.
4 
5 Many Muds use 'details' to add more flavour. 'Details' are items which can be looked at, but are not implemented as own objects, but instead simulated by the environment. Lets assume that the function
6 
7 ~~~{.c}
8 AddDetail(string keyword, string|closure desc)
9 
10 ~~~
11 adds the detail 'keyword' to the room, which, when look at, returns the string 'desc' resp. the result of the execution of closure 'desc' as the detail description to the player.
12 
13 Now imagine that one wants to equip a room with magic runes, which read as `'Hello <playername>!\n"` when looked at. Obviously
14 
15 ~~~{.c}
16 AddDetail("runes", sprintf("Hello %s!\n", this_player()->QueryName()));
17 
18 ~~~
19 is not sufficient, as the `this_player()` is executed to early and just once: for the player loading the room.
20 
21 The solution is to use closures. First, the solution using lfun-closures:
22 
23 ~~~{.c}
24 private string _detail_runes () {
25  return sprintf("Hello %s!\n", this_player()->QueryName());
26 }
27 ...
28 AddDetail("runes", #'_detail_runes);
29 
30 ~~~
31 or with an inline closure:
32 
33 ~~~{.c}
34 AddDetail(
35  "runes",
36  (: sprintf("Hello %s!\n", this_player()->QueryName()) :)
37 );
38 
39 ~~~
40 Simple? Here is the same code, this time as lambda-closure:
41 
42 ~~~{.c}
43 AddDetail(
44  "runes",
45  lambda(
46  0,
47  ({
48  #'sprintf,
49  "Hello %s!\n",
50  ({#'call_other, ({#'this_player}), "QueryName" })
51  })
52  )
53 );
54 
55 ~~~
56 Why the extra `({ })` around `#'this_player`? `#'this_player` alone is just a symbol, symbolizing efun::this_player(), but efun::call_other() needs an object as first argument. Therefore, the `#'this_player` has to be interpreted as function to evaluate, which is enforced by enclosing it in `({ })`. The same reason also dictates the enclosing of the whole `#'call_other` expression into `({ })`. Note also the missing `#'return`: it is not needed. The result of a lambda-closure is the last value computed.
57 
58 Another example: Task is to reduce the HP of every living in a room by 10, unless the result would be negative. Selecting all livings in a room is simply:
59 
60 ~~~{.c}
61 filter(all_inventory(room), #'living)
62 
63 ~~~
64 The tricky part is to reduce the HP. Again, first the lfun-closure solution:
65 
66 ~~~{.c}
67 private _reduce_hp (object liv) {
68  int hp;
69  hp = liv->QueryHP();
70  if (hp > 10){
71  liv->SetHP(hp-10);
72  }
73 }
74 ...
75 
76 map( filter(all_inventory(room), #'living), #'_reduce_hp)
77 
78 ~~~
79 or as an inline closure:
80 
81 ~~~{.c}
82 map(
83  filter(all_inventory(room), #'living),
84  (:
85  int hp;
86  hp = liv->QueryHP();
87  if (hp > 10){
88  liv->SetHP(hp - 10);
89  }
90  :)
91 );
92 
93 ~~~
94 Both efun::filter() and efun::map() pass the actual array item being filtered/mapped as first argument to the closure.
95 
96 Now, the lambda-closure solution:
97 
98 ~~~{.c}
99 map(
100  filter(all_inventory(room), #'living),
101  lambda(
102  ({ 'liv }),
103  ({
104  #', , // not obvious if this is one arg or two?
105  ({ #'=, 'hp, ({#'call_other, 'liv, "QueryHP" }) }),
106  ({
107  #'?,
108  ({#'>, 'hp, 10 }),
109  ({ #'call_other, 'liv, "SetHP", ({#'-, 'hp, 10 }) })
110  })
111  })
112  ) // of lambda()
113 );
114 
115 ~~~
116 It is worthy to point out how local variables like 'hp' are declared in a lambda-closure: not at all. They are just used by writing their symbol 'hp . Same applies to the closures parameter 'liv .
117 
118 The lambda-closure solution is not recommended for three reasons: it is complicated, does not use the powers of efun::lambda(), and the efun::lambda() is recompiled every time this statement is executed!
119 
120 So far, lambda-closures seem to be just complicated, and in fact: they are. Their powers lie elsewhere.
121 
122 Imagine a computation, like for skill resolution, which involves two object properties multiplied with factors and then added.
123 
124 The straightforward solution would be a function like:
125 
126 ~~~{.c}
127 int Compute (object obj, string stat1, int factor1, string stat2,
128  int factor2)
129 {
130  return (
131  call_other(obj, "Query"+stat1) * factor1 +
132  call_other(obj, "Query"+stat2) * factor2
133  );
134 }
135 
136 ~~~
137 Each call to `Compute()` involves several operations (computing the function names and resolving each efun::call_other()) which in fact need to be done just once. Using lambda-closures, one can construct and compile a piece of code which behaves like a `Compute()` tailored for a specific stat/factor combination:
138 
139 ~~~{.c}
140 closure ConstructCompute (object obj, string stat1, int factor1,
141  string stat2, int factor2)
142 {
143  mixed code;
144 
145  // Construct the first multiplication.
146  // The symbol_function() creates a symbol for the
147  // lfun 'Query<stat1>', speeding up later calls.
148  // Note again the extra ({ }) around the created symbol.
149 
150  code = ({#'*, ({ symbol_function("Query"+stat1, obj) }), factor1 });
151 
152  // Construct the second multiplication, and the addition
153  // of both terms.
154 
155  code = ({
156  #'+,
157  code,
158  ({#'*, ({ symbol_function("Query"+stat2, obj) }), factor2 })
159  });
160 
161  // Compile the code and return the generated closure.
162  return lambda(0, code);
163 }
164 
165 ~~~
166 Once the closure is compiled,
167 
168 ~~~{.c}
169 str_dex_fun = ConstructCompute(obj, "Str", 10, "Dex", 90);
170 
171 ~~~
172 it can be used with a simple `funcall(str_dex_fun)`.
173 
174 ## When to use which closure? #
175 
176 First, a closure is only then useful if it needn't to live any longer than the object defining it. Reason: when the defining object gets destructed, the closure will vanish, too.
177 
178 Efun-, lfun- and inline closures should be used where useful, as they mostly do the job and are easy to read. The disadvantage of lfun- and inline closures is that they make a efun::replace_program() impossible - but since such objects tend to not being replaceable at all, this is no real loss.
179 
180 Lambda closures are needed if the actions of the closure are heavily depending on some data available only at runtime, like the actual inventory of a certain player.
181 
182 If you use lfun-closures and find yourself shoving around runtime data in arguments or (gasp!) global variables, it is time to think about using a lambda-closure, compiling the value hard into it.
183 
184 The disadvantages of lambda closures are clear: they are damn hard to read, and each efun::lambda() statement requires extra time to compile the closure.
