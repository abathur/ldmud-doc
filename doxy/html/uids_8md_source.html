1 userids {#driver_topics_uids}
2 =============================
3 Every object in the mud is attributed with a user-id 'uid': a string which associates the object with a certain 'user' (aka 'wizard' or 'creator', though it is not limited to that). The uid can be 0, which internally is the default-uid.
4 
5 The uid serves a dual purpose: on the on hand it is used to gather statistics about the various groups of objects (in the famous 'wizlist'), on the other hand the uid can come in handy in the implementation of security systems.
6 
7 The uid of an object is assigned at its creation through the driver hooks @ref driver_hook_load_uids "H_LOAD_UIDS" for loaded objects, and @ref driver_hook_clone_uids "H_CLONE_UIDS" for cloned objects, and can't be changed afterwards.
8 
9 The uid of an object can be queried with the efun::getuid() (resp. efun::creator() in compat-mode).
10 
11 Every object also has a second string attribute, the 'effective userid' or 'euid', which also may be 0. This value was intended to implement a security system based on difference between theoretical and effective permissions. Since the effectiveness of this system is doubtful, the driver enforces such a use only as an option.
12 
13 As uids, euids are assigned at an objects creation through the two aformentioned driverhooks. They can be queried with efun::geteuid() and changed with efun::seteuid(). Calls to the latter are verified by master::valid_seteuid().
14 
15 Additionally objects can impose their uid onto an other objects euid with the efun::export_uid().
16 
17 If the driver is run in 'strict euids' mode, euids are taken more seriously than being just another attribute:
18 
19 - all objects must have a non-0 uid.
20 - objects with a 0 euid can't load or clone other objects.
21 - the backbone uid as returned by master::get_bb_uid() must not be 0.
22 
23 Userids are assigned at the time of the creation of an object by calling the driverhooks @ref driver_hook_load_uids "H_LOAD_UIDS" and @ref driver_hook_clone_uids "H_CLONE_UIDS":
24 
25 ~~~{.c}
26 mixed <load_uids closure> (string objectname)
27 mixed <clone_uids closure>(object blueprint, string objectname)
28 
29 ~~~
30 When an object is newly loaded, the @ref driver_hook_load_uids "H_LOAD_UIDS" hook is called with the object name as argument.
31 
32 When an object is cloned, the @ref driver_hook_clone_uids "H_CLONE_UIDS" hook is called with the blueprint object as first and the clone's designated name as second argument.
33 
34 In both cases the new object already exists, but has 0 uids.
35 
36 For the result, the following possibilities exist (`<num>` is a non-zero number, `<no-string>` is anything but a string):
37 
38 ~~~{.c}
39 "<uid>" -> uid = "<uid>", euid = "<uid>"
40 ({ "<uid>", "<euid>" }) -> uid = "<uid>", euid = "<euid>"
41 ({ "<uid>", <no-string> }) -> uid = "<uid>", euid = 0
42 
43 ~~~
44 If strict-euids is not active, the following results are possible, too:
45 
46 ~~~{.c}
47 <num> -> uid = 'default', euid = 0
48 ({ <num>, "<euid>" }) -> uid = 'default', euid = "<euid>"
49 ({ <num>, <no-string> }) -> uid = 'default', euid = 0
50 
51 ~~~
52 Slightly different rules apply to the (e)uid of the master. The master's (e)uid is determined by a call to master::get_master_uid():
53 
54 ~~~{.c}
55 "<uid"> -> uid = "<uid>", euid = "<uid>"
56 
57 ~~~
58 In non-strict-euids mode, more results are possible:
59 
60 ~~~{.c}
61 0 -> uid = 0, euid = 0
62 <num> -> uid = 'default', euid = 0
63 
64 ~~~
65 If your uids are in general based on filenames, it is wise to return a value here which can not be legally generated from any filename. OSB for example uses 'ze/us'.
66 
67 The masters uid is determined only on startup this way, at runtime the uids of a reloaded master determined as for every object by a call to the appropriate driver hooks.
68
