11 Clone a new object from definition <name>, or alternatively from
12 the object <template>. In both cases, the new object is given
13 an unique name and returned.
14 
15 The original used for cloning, called blueprint, should not be
16 used in the system, just for cloning. The cloned objects
17 contain only the data but the blueprint also the function code.
18 The blueprint is the one without a unique number at the end of
19 the object's name. The clone_object() function never
20 returns a blue print.
21 
22 If the <name> or <template> designates a cloned object itself,
23 the system looks up the blueprint object _by >>name_<<.
24 
25 If the blueprint exists and has a heart_beat(), clone_object()
26 turns it off.
27 
28 Note that the pathname must be complete, which means there are no
29 relative paths allowed.
30 
31 If strict euids are enforced, the cloning object must have
32 a non-zero euid.
33 
34 -- Variable Initialization --
35 
36 In general, variables are initialized for blueprints and clones alike
37 with a call to the internal lfun __INIT().
38 
39 However, if #pragma share_variables is in effect (either explicitely
40 given in the source or implicitely as runtime option), the values for
41 a clone's uninitialized variables are taken from the _current_
42 variables of the object's blueprint.
43 
44 In the absence of share_variables, variables without explicit
45 initializers are initialized to 0.
