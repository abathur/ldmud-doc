1 The Mud Client Compression Protocol {#driver_topics_mccp}
2 =========================================================
3 Information and code taken from the @ref driver_topics_mccp "MCCP" Homepage @subpage driver_http:__www.randomly.org_projects_MCCP_ "http://www.randomly.org/projects/MCCP/"
4 
5 @ref driver_topics_mccp "MCCP" is implemented as a Telnet option [RFC854, RFC855]. The server and client negotiate the use of @ref driver_topics_mccp "MCCP" as they would any other telnet option. Once agreement has been reached on the use of the option, option subnegotiation is used to determine acceptable compression methods to use, and to indicate the start of a compressed data stream.
6 
7 If the driver is compiled with @ref driver_topics_mccp "MCCP" Support there is a define @ref driver_topics_predefined "__MCCP__".
8 
9 The driver currently supports both versions of mccp. If your mud has a @ref driver_hook_no_echo "H_NOECHO" hook you have to find out if the client supports mccp. Without this hook you still have to start neogotiation.
10 
11 All sub-negotiation is done by efun::start_mccp_compress() and efun::end_mccp_compress() whether you have this hook or not.
12 
13 @par note: when the client uses compressions all binary_message calls are executed with flag=3. This is because writing to the socket would disturb zlib stream.
14

22 Initiating @ref driver_topics_mccp "MCCP" without @ref driver_hook_no_echo "H_NOECHO" hook:
23 
24 ~~~{.c}
25 if(!query_mccp()){
26  binary_message(({ IAC, WILL, TELOPT_COMPRESS2 }),1)
27  binary_message(({ IAC, WILL, TELOPT_COMPRESS }),1)
28 }
29 
30 ~~~
31 the driver will parse the client's answers and start compression. (The connection might already be compressed, because although the documentation says clients should not negotiate from themselves, zmud e.g. does.)
32 
33 You can start and stop compression manually by efuns when you are sure client supports compression :)
34 
35 Initiating @ref driver_topics_mccp "MCCP" compression with @ref driver_hook_no_echo "H_NOECHO" hook:
36 
37 ~~~{.c}
38 If your mudlib uses the :hook:`H_NOECHO` driver-hook you decided to do
39 all the negotiation by yourself:
40 
41 Server Commands
42 IAC WILL COMPRESS indicates the sender supports version 1 of the
43  protocol, and is willing to compress data it sends.
44 
45 IAC WILL COMPRESS2 indicates the sender supports version 2, and is
46  willing to compress data it sends.
47 
48 IAC WONT COMPRESS indicates the sender refuses to compress data using
49  version 1.
50 
51 IAC WONT COMPRESS2 indicates the sender refuses to compress data
52  using version 2.
53 
54 Client Commands
55 IAC DO COMPRESS indicates the sender supports version 1 of the
56  protocol, and is willing to decompress data
57  received.
58 
59 IAC DO COMPRESS2 indicates the sender supports version 2 or above,
60  and is willing to decompress data received.
61 
62 IAC DONT COMPRESS indicates the sender refuses to support version 1.
63  If compression was previously negotiated and is
64  currently being used, the server should terminate
65  compression.
66 
67 IAC DONT COMPRESS2 indicates the sender refuses to support version 2.
68  If compression was previously negotiated and is
69  currently being used, the server should terminate
70  compression
71 
72 After you found out whether the client supports mccp or not you can
73 start compression with start_mccp_compress(TELOPT_COMPRESS2) or
74 start_mccp_compress(TELOPT_COMPRESS). ( you could start it without
75 checking but some players would protest :) )
76 
77 ~~~
