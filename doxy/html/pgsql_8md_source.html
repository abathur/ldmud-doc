1 pgsql {#driver_concepts_pgsql}
2 ==============================
3 On hosts with the PostgreSQL package installed, the driver can be configured to interface with the PostgreSQL database. If that is done, the driver defines the macro @ref driver_driver_predefined "__PGSQL__" for LPC programs and activates a number of related efuns.
4 
5 ## SECURITY #
6 
7 All SQL efuns (unless execute by the master or the simul-efun object) trigger a privilege_violation ("pgsql", "<efun_name>"). If a more fine-grained control is desired, overload the individual efuns with a nomask simul_efun.
8 
9 Authored by Florian Heinz and others.
10 
11 @usage{
12 The interface to the PostgreSQL database is implemented through the concept of a controlling object: when opening a database connection, the LPC code has to provide a callback function. The object this function is bound to is the controlling object: all queries to the database will be issued by this object, and the responses will be sent to the callback function.
13 The interface is also asynchronous: the pg_query() efun just queues the query with the database connection, and returns immediately. When the database has finished working the query, the callback function is called with the results.
14 The callback function can be defined by name or by closure, and can be defined with extra parameters:
15 #include <pgsql.h>
16 
17 void <callback>(int type, mixed ret, int id [, mixed extra...])
18 
19  <type> is the type of the call, <id> identifies the query
20  for which this call is executed:
21 
22  PGRES_TUPLES_OK: <ret> is the result from a query.
23  It is either a mapping (field name as
24  key, indexing <n> values for n returned
25  tuples), or an array of arrays (one per
26  row).
27 
28  PGRES_COMMAND_OK: <ret> is a string which contains the
29  server response (e.g. on INSERT or DELETE)
30 
31  PGRES_BAD_RESPONSE,
32  PGRES_NONFATAL_ERROR,
33  PGRES_FATAL_ERROR: ret is the error-string
34 
35 
36 void <callback>(int type, mixed ret [, mixed extra...])
37 
38  <type> is the type of the call, which is not related to a
39  specific query:
40 
41  PGCONN_SUCCESS: The database-connection was established,
42  <ret> is a dummy string.
43  PGCONN_FAILED: The database-connection failed, <ret> is
44  the error message.
45  The first message to the callback after a call to
46  pg_connect() is always one of these two.
47 
48  PGRES_NOTICE: <ret> is a informational text.
49 
50  PGCONN_ABORTED: If the connection to the backend fails
51  we try to re-establish (reset) it. If the
52  reset fails, the connection is closed and
53  this value is returned. Consider the
54  connection gone and don't try to close or
55  otherwise operate further on it.
56  <ret> is a dummy string.
57 }
