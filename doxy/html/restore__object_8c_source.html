11 Restore values of variables for current object from the file <name>,
12 or directly from the string <str>.
13 
14 To restore directly from a string <str>, the string must begin
15 with the typical line "#x:y" as it is created by the save_object()
16 efun.
17 
18 When restoring from a file, the name may end in ".c" which is stripped
19 off by the parser. The master object will probably append a .o to the
20 <name>. The validity of the filename is checked with a call to
21 check_valid_path().
22 
23 Return 1 on success, 0 if there was nothing to restore.
24 
25 Variables that has the type modifer 'nosave' will not be restored.
26 Example: nosave int xxx;
27 
28 Lfun, variable and simul_efun closures are restored only if they
29 can be found (this excludes closures of private lfuns as well) - if
30 not, they are restored as value '0'.
31 
32 If inheritance is used, then it might be possible that a
33 variable will exist with the same name in more than one place,
34 and thus appear in the save file multiple times. When
35 restoring, the variables are restored in the order they are
36 found in the inheritance tree. A good practice is to have
37 verbose and unique name on non-static variables, which also
38 will make it more easy to read or patch the save file
39 manually.
40 
41 Restoring directly from a string was added in LDMud 3.2.8 and
42 may be moved in future into a separate efun.
43 LDMud 3.2.9 added the restoring of non-lambda closures, symbols,
44 and quoted arrays, using a new savefile format version.
45 LDMud 3.5.0 added the possibility to restore version 2 with its higher
46 float precision.
