1 inheritance {#driver_topics_inheritance}
2 ========================================
3 Have you noticed how many objects in the system have the same functionality in common? Let's look at rooms for instance, they all have the ability to host people and provide commands. It's not that every room is programmed with the same basic functions again and again, rather it will use a model room and then make some special changes to it. That doesn't work by copying the file.. Ouch! Don't replicate code! But by putting a tiny inherit declaration:
4 
5 ~~~{.c}
6 inherit "<model-class>";
7 
8 ~~~
9 at the beginning of your new file. This must come before any local variables or functions. Once inherited your class will behave just like the model class, because all the public methods are available to the outside world. Now it is in your hands to change such an inherited behaviour. You have the following tools to do so:
10 
11 ## Access to variables #
12 
13 It is one of the best design decisions in LPC that variables are not accessible from outside, but you can use inherited variables just as if they were your own. Modifiers apply however.
14 
15 ## Method overloading #
16 
17 You can simply rewrite a method that is also defined in the model class, and thus change how it behaves:
18 
19 ~~~{.c}
20 int method_that_also_exists_in_the_model() {
21  <your new code>
22 }
23 
24 ~~~
25 Contrary to other languages in LPC method overloading only matches the name of the method, so even by changing the amount and type of parameters you will mask out the original version of the method. You can even apply other modifiers to it as the original.
26 
27 ## Example #
28 
29 Let's imagine very simple food in a file called "/the/food.c":
30 
31 ~~~{.c}
32 // unless "modified" variables are accessible by inheritors
33 int vitamins = 10;
34 
35 // please overload this function with your own description
36 public short() { return "something edible"; }
37 
38 // let's do some standard action for food
39 public consume() {
40  this_player() -> nourish(vitamins);
41  destruct(this_object());
42 }
43 
44 ~~~
45 And now someone else decides to do some italian cooking in a file called "/the/fusilli.c":
46 
47 ~~~{.c}
48 inherit "/the/food";
49 
50 // we have our own variables.
51 int gone_cold = 0;
52 
53 // and we simply redefine the short() function to replace it
54 public short() {
55  // description changes depending on gone_cold
56  return "a "+( gone_cold ? "stinking" : "steaming" )+" plate of fusilli";
57 }
58 
59 // we have a new function to make food go cold
60 private deteriorate() {
61  gone_cold = 1;
62  write("The fusilli have gone cold.\n");
63 }
64 
65 // assume this gets called at creation
66 private create() {
67  // we can access the variable we inherited from food.c
68  vitamins = 44; // tomato has plenty of vitamins
69 
70  // go cold in 5 minutes
71  call_out( #'deteriorate, 5 * 60 );
72 }
73 
74 // we can overload the function even with new parameters
75 public consume(how) {
76  // fetch the name of the person, or use "Someone"
77  string name = this_player() -> name() || "Someone";
78 
79  if (!gone_cold) {
80  write("You enjoy a delicious plate of fusilli.\n");
81  say(name +" guzzles a plate of hot fusilli.\n");
82  }
83  else if (how == "quickly") {
84  write("You eat the fusilli so quickly you hardly notice they have gone cold.\n");
85  say(name +" wolfs down a plate of cold fusilli.\n");
86  }
87  else {
88  write("You eye the plate and wonder if you really feel like eating cold fusilli.\n");
89  return; // don't eat
90  }
91 
92  // and here comes the most important part:
93  // we execute consume() from food.c, so we
94  // actually inherit its behaviour.
95  ::consume();
96 }
97 
98 ~~~

102 ### Doing multiple inheritance #
103 
104 While the Java(TM) language has so-called interfaces as a kludge, LPC doesn't need them as it supports real multiple inheritance. A very powerful feature, it lets you combine the behaviour of several classes into a new one. Simply put several lines of inherit declarations underneath each other. If you have name collisions in the namespace of inherited methods, you will have to address them explicitely with a `"the/file"::method(args)` syntax.
105 
106 ## Advanced Example #
107 
108 ~~~{.c}
109 inherit "foo";
110 inherit "bar";
111 inherit "baz";
112 inherit "ball";
113 
114 reset() {
115  "ba?"::reset();
116  // calls bar::reset() and baz::reset()
117 
118  "ba*"::reset();
119  // calls bar::reset(), baz::reset() and ball::reset()
120 
121  "*"::reset();
122  // calls every inherited reset() function.
123 
124  "ball"::rejoice("Listen to italectro today!");
125  // only explicit filename of model class allows
126  // passing arguments to the inherited method
127 }
128 
129 ~~~
